# Agent Skill Synthesis Governance OpenSpec Design

## Overview

This change adds governed agent skill synthesis after the wiki kernel is semantically stable:

```text
evidence
  -> context economy
  -> AI distill
  -> wiki curate
  -> wiki semantic review
  -> review/promote wiki
  -> skill signal clustering
  -> skill proposal synthesis
  -> skill semantic review
  -> audited skill review/promote
  -> stable skills/
```

M19.1 completes the wiki semantic gate. M19.2 adds the skill lane.

## Source-Level Borrowing

The implementation borrows specific mechanisms from the reference repos, not their whole runtimes:

- Hermes `agent/background_review.py`: dedicated background review prompt, memory/skill-only tool boundary, update-before-create ladder, and anti-pattern list.
- Hermes `tools/skill_manager_tool.py`: `SKILL.md` frontmatter validation, support-file directories, skill-name validation, and inventory lookup shape.
- Hermes `tools/skill_provenance.py`: write-origin provenance as the reason PraxisBase needs explicit promotion audit metadata.
- Hermes `tools/skill_usage.py` and `agent/curator.py`: usage-aware curation, pinned/archive concepts, umbrella consolidation, and tool-call audit reports.
- OpenHuman `docs/AGENT_SELF_LEARNING.md`, `learning/candidate.rs`, and `learning/stability_detector.rs`: candidate buffers, cue families, stability scoring, active/provisional separation, and class budgets.
- OpenHuman `skills/inject.rs`, `context/tool_result_budget.rs`, and `context/microcompact.rs`: bounded skill/context injection and prompt economy.

PraxisBase deliberately does not borrow Hermes' direct stable-skill mutation authority or OpenHuman GPL/vendor runtime code.

## M19.1 Wiki Gate Prerequisite

Before stable skill promotion depends on wiki pages, PraxisBase must finish the existing semantic-review tasks:

- daily reports include semantic review counts;
- personal auto-promotion requires a passing semantic review for new wiki pages;
- semantic reviewer input uses source summaries, distilled summaries, candidate bodies, and provenance excerpts;
- AgentMemory sidecar hits do not count as promotion evidence unless ingested;
- site/review UI renders semantic review decisions and reasons;
- bad-example regressions cover one-off smoke reports, task-runner fragments, merge-worthy replay fragments, and raw-ish summaries.

## Skill Candidate Authority

Skill synthesis has two authority levels:

- candidate: generated by AI and reviewable;
- stable skill: promoted under `skills/**` with audit trail.

Both personal and team modes may create candidates. Neither mode may write stable `skills/**` directly from distill, daily, or skill synthesis.

Personal mode requires user audit before stable skill promotion. Team mode requires human/Git review before stable skill promotion.

## Signal Sources

Skill synthesis consumes only already-normalized, provenance-bearing inputs:

- `DistilledExperience.skill_candidate`;
- stable or promoted wiki procedures, known fixes, pitfalls, and preferences;
- repeated verified outcomes from source summaries;
- existing stable skill inventory for update decisions.

Raw transcripts and raw logs are never skill synthesis inputs.

## Signal Gate

PraxisBase SHALL reject or route away weak skill signals:

- one-off task narratives;
- exact run ids, PR ids, error strings, or feature codenames as skill topics;
- environment setup failures unless the durable lesson is the fix pattern;
- negative claims such as "tool X is broken";
- transient resolved errors without reusable retry/fallback guidance;
- raw transcript/log copies;
- unsupported generic advice;
- privacy or scope violations.

## Stability And Clustering

PraxisBase SHALL group repeated skill signals before proposal synthesis.

The first implementation is file-first. It may write reports under `.praxisbase/reports/skill-synthesis/` and candidate records under `.praxisbase/inbox/proposals/`.

The clustering model includes:

- cue family;
- evidence refs and hashes;
- observed time;
- confidence;
- source count;
- scope;
- related wiki pages;
- existing skill match.

Repeated verified signals increase eligibility. Low-confidence singleton signals stay out of the primary review queue.

## Stable Skill Inventory

Before proposing a new skill, PraxisBase SHALL load existing stable `skills/**/SKILL.md` inventory and score possible update targets. Inventory includes path, slug, frontmatter name/description/scope, headings, `When To Use`, `Procedure`, `Pitfalls`, `Provenance`, related wiki links, and optional usage metadata.

Match classes:

- strong: same normalized domain plus overlapping trigger/procedure entities;
- medium: description or `When To Use` covers the class but procedure differs;
- weak: same tool/product name only;
- none: no shared domain, trigger, or procedure.

Strong and medium matches are provided to the proposer as update targets. Multiple strong matches require `needs_human` or merge/update; they must not produce a new sibling skill.

## Skill Decision Ladder

The proposer SHALL choose the earliest fitting action:

1. update a loaded or explicitly referenced stable skill;
2. update an existing class-level umbrella skill;
3. add a support file under an existing umbrella;
4. create a new class-level umbrella skill.

New skill creation is last resort. Candidate names must be class-level and stable.

## Candidate Shape

Each candidate SHALL include:

- action: `skill_create`, `skill_update`, or `skill_support_file`;
- target path under `skills/**`;
- source refs, source hashes, evidence ids, and source count;
- related wiki page paths;
- ladder choice;
- review hint;
- generated body or patch.

`SKILL.md` candidates SHALL include these sections:

- `## When To Use`
- `## Procedure`
- `## Verification`
- `## Pitfalls`
- `## Do Not Use When`
- `## Related Wiki Pages`
- `## Provenance`

Support file candidates SHALL write only under `references/`, `templates/`, or `scripts/`, and SHALL update `SKILL.md` with a pointer.

## Skill Semantic Review

PraxisBase SHALL review skill candidates with a semantic skill reviewer that is distinct from the wiki reviewer.

The reviewer returns:

- `approve_candidate`;
- `revise`;
- `merge_or_update_existing`;
- `reject`;
- `needs_human`.

Approval requires a class-level topic, actionable procedure, reusable trigger, safe future-agent behavior, and provenance-backed evidence. Approval does not directly promote the skill.

## Review And Promotion

Stable skill promotion requires an audit record with proposal id, candidate id, target path, reviewer, semantic review id, source hashes, decision, and created timestamp.

Promotion rules:

- audit decision must be `approved`;
- proposal id, candidate id, target path, and source hashes must match the candidate;
- personal promotion requires a user reviewer;
- team promotion requires Git/human review metadata or configured team policy;
- automation may mark candidates reviewed but cannot promote brand-new stable skills by default.

Personal default:

- candidate generation allowed;
- LLM skill review allowed;
- stable promotion requires user audit;
- no default auto-promotion of brand-new skills.

Team default:

- candidate generation allowed for team-safe evidence;
- LLM skill review allowed;
- stable promotion requires Git/human review;
- personal/private evidence is blocked from team skills.

## Site And CLI

Add skill commands:

```bash
praxisbase skill synthesize --mode personal --review --json
praxisbase skill review --json
praxisbase skill promote --proposal <id> --json
praxisbase skill curate --dry-run --json
praxisbase skill export --agent codex --json
```

The generated site SHALL show skill candidate counts, cards, reviewer decisions, score, reason, scope, provenance, and next commands.

## Real Smoke Bar

A real personal smoke is not successful merely because files are written. It must show separate counts for signals, rejected or low-stability signals, clusters, reviewed candidates, rejected candidates, human-required candidates, and promoted stable skills. The site must show reviewed candidates as the primary queue. Stable `skills/**` must remain unchanged until audited promotion. Candidate bodies must be synthesized, provenance-backed, and free of raw transcripts, secrets, private paths, and team-unsafe personal material.

## Failure Modes

- AI unavailable: write a report and skip candidate creation unless degraded mode is explicit.
- Reviewer unavailable: candidate requires human review and cannot be promoted automatically.
- Existing skill ambiguity: route to `needs_human`.
- Privacy/scope uncertainty: route to privacy or human-required queue.
- Support file path outside allowed directories: reject.
- Candidate body missing required sections: revise once, then reject or human-required.

## Observability

Reports distinguish:

- raw signals;
- rejected or low-stability signals;
- clustered signals;
- synthesized candidates;
- semantic reviews;
- rejected candidates;
- human-required candidates;
- promoted stable skills.

The primary review queue is reviewed candidates, not raw signals.
